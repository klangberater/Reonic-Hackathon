# Lumen — the home-energy assistant Enpal should ship

**Reonic / Enpal hackathon entry.** Lumen turns a solar home's raw telemetry into one calm
answer to the only question that matters: *"what should I do right now to use more of my own
sun and pay less?"* — and then does it for you.

It is a native iOS app on top of a deterministic energy-planning backend, with a grounded
LLM assistant layered on top. The numbers come from real physics and real data; the words
come from the model. Nothing is faked for the demo: the planner, the savings, the anomaly
detection and the schedule are all computed from a coherent year of 15-minute telemetry.

> **Live API:** `https://getfletcher.ai/api` · **iOS:** SwiftUI (TestFlight) · **Backend:** TypeScript/Express

## What it does

Three product surfaces, each a "demo moment":

### 1. The glance (Home screen)
One screen that answers "is my home OK and what's it doing?" — a plain-language verdict
("Running on free solar — sending 7.7 kW to the grid"), a live power-flow detail, a
month-to-date money line ("on track to **earn** €15 this month"), and a tappable list of
flexible devices. A **"Needs a look"** card appears only when something is genuinely wrong.

### 2. Plan my day (swipe right)
Pick energy-heavy tasks (charge car, run dishwasher/washing machine/dryer, hot-water or
heating boost), set a "done by" time for each, and get a **schedule that runs them on as much
solar as possible**. A hero timeline draws the day's solar curve with task blocks slotted
under it, split-shaded by how much of each run is your own solar vs grid. Toggle
**Cheapest / Greenest / Soonest**, nudge a block, or re-plan. A summary chip shows
"% solar · € saved · kg CO₂".

### 3. Ask anything (grounded chat)
A natural-language assistant that **only speaks numbers it computed**. It runs an
OpenAI function-calling loop over the same planner tools the app uses, so "when should I run
the dishwasher?" or "why is my heat-pump bill so high?" get answered from live data — and the
heat-pump answer is backed by real **weather-normalised anomaly detection**, not a guess.

## Architecture at a glance

```
┌──────────────────────────┐         HTTPS / JSON          ┌───────────────────────────────────┐
│        iOS app            │  ───────────────────────────▶ │   Backend (Node + TypeScript)      │
│        (SwiftUI)          │   /now /money /devices         │                                    │
│                           │   /optimize_load /plan_day     │  REST API (Express)  ── server.ts  │
│  RootPager (swipe)        │   /insights /chat ...          │       │                            │
│   ├─ HomeView  (glance)   │ ◀───────────────────────────  │       ├─ planner     optimizeLoad  │
│   ├─ PlanDayView (plan)   │        JSON responses          │       ├─ day planner planDay       │
│   └─ ChatView  (assistant)│                                │       ├─ anomaly     anomaly.ts    │
│                           │                                │       ├─ money       money.ts      │
│  ClockStore (shared now)  │                                │       ├─ ledger      ledger.ts     │
└──────────────────────────┘                                │       └─ chat        openaiChat.ts │
                                                             │              │  (OpenAI gpt-4o)    │
                                                             │       data.ts │ read models views  │
                                                             └───────────────┼────────────────────┘
                                                                             ▼
                                              data/  — 4 homes × 35,040 × 15-min records (2026)
                                              generated by scripts/transform_dataset.py
                                              from enpal-track-dataset/ + real Munich forecast
```

- **No database.** The "source of truth" is a coherent, physically-balanced JSON dataset; the
  only mutable state is an in-memory committed-loads ledger.
- **Tools own the numbers, the LLM owns the words.** Every figure the assistant says is the
  return value of a deterministic function call.
- **Deploy is one push.** Push to `main` → GitHub Actions → SSH → build → restart systemd →
  smoke-test. iOS ships via TestFlight.

## Tech stack

| Layer | Technology | Notes |
|------|-----------|-------|
| iOS app | **Swift 5 · SwiftUI · iOS 17** | No third-party Swift packages. `URLSession` async networking, `UserNotifications` for reminders. |
| iOS project | **XcodeGen** | `ios/project.yml` is the source of truth; the `.xcodeproj` is generated (gitignored). |
| Backend runtime | **Node.js 20+ · TypeScript 5.5** | CommonJS, compiled with `tsc`. |
| Web framework | **Express 4** | Thin REST layer; endpoints map 1:1 to product surfaces. |
| Tests | **`node:test`** (built-in) | Zero test dependencies; run with `npm test`. |
| Assistant | **OpenAI Chat Completions API** (`gpt-4o`) | Function-calling loop over the planner tools. |
| Data pipeline | **Python 3** | `scripts/transform_dataset.py`; pulls a real **Open-Meteo** Munich forecast. |
| Hosting | **Ubuntu · systemd · nginx + Certbot (TLS)** | `getfletcher.ai`, backend bound to `127.0.0.1:8090`. |
| CI/CD | **GitHub Actions** | `.github/workflows/deploy.yml` (backend), `deploy/testflight.sh` (iOS). |

Full inventory and rationale: **[docs/ARCHITECTURE.md](docs/ARCHITECTURE.md)**.

## Quick start

Prerequisites: **Node 20+**, **Python 3** (only to regenerate data), **Xcode 16 + XcodeGen**
(only for the app). macOS for the iOS app; the backend runs anywhere Node does.

### 1. Backend

```bash
cd backend
npm install
npm run build        # tsc → dist/
npm start            # → reonic-backend on http://127.0.0.1:8090
```

Smoke-test it:

```bash
curl -s localhost:8090/health
curl -s "localhost:8090/now?household=HH-1001&clock=summer" | python3 -m json.tool
```

Dev loop (rebuild + restart on change):

```bash
npm run dev
```

The assistant endpoint (`POST /chat`) additionally needs `OPENAI_API_KEY` in the environment;
every other endpoint works with no secrets. See [Configuration](#configuration).

### 2. Dataset

The committed `data/` is ready to use — you do **not** need to regenerate it. To rebuild it
from the raw organizer data (and re-pull the Munich forecast):

```bash
python3 scripts/transform_dataset.py     # enpal-track-dataset/ → data/
```

Why a transform exists (the raw data's temperature/heat-pump channels are seasonally
inverted) and exactly what it rebuilds: **[data/README.md](data/README.md)**.

### 3. iOS app

```bash
brew install xcodegen           # one-time
cd ios && xcodegen generate     # project.yml → Lumen.xcodeproj
open Lumen.xcodeproj            # ⌘R to run
```

- **Simulator:** runs as-is — the app points at the public HTTPS API, no local backend needed.
- **Local backend instead:** set `Config.baseURL` to `http://127.0.0.1:8090` (add an ATS
  exception for cleartext localhost) and run the backend from step 1.
- **Real device / TestFlight:** see **[deploy/TESTFLIGHT.md](deploy/TESTFLIGHT.md)**.

> The `.xcodeproj` is generated and gitignored — **always run `xcodegen generate` after
> pulling** so new Swift files are in the target.

More: **[ios/README.md](ios/README.md)**.

## Configuration

### Backend (environment variables)

| Variable | Default | Purpose |
|----------|---------|---------|
| `PORT` | `8090` | HTTP port (bound to `127.0.0.1`). |
| `OPENAI_API_KEY` | — | Required **only** for `POST /chat`. Without it, `/chat` returns `503`; everything else works. |
| `OPENAI_MODEL` | `gpt-4o` | Model for the assistant. |
| `DEMO_CLOCK` | `summer` | Default clock when a request omits `?clock=`. `summer` = live wall clock; `winter` = fixed scenario. |
| `CHAT_TOKEN` | — | Optional shared secret; if set, `/chat` requires header `x-lumen-token`. |

### Request parameters (all endpoints)

- `?household=` — one of `HH-1001`…`HH-1004` (default `HH-1001`, the "hero" home).
- `?clock=` — `summer` (live real time) or `winter` (fixed Jan morning for the anomaly demo).
- `?at=` — ISO timestamp override for scripted demos/tests (bypasses the clock).

### iOS

`ios/Sources/Config.swift` — `baseURL` (default `https://getfletcher.ai/api`),
`defaultHousehold`, optional `chatToken`.

## Testing

Deterministic backend logic (planner objectives, device library, day-plan orchestration,
savings, staggering) is covered by `node:test`:

```bash
cd backend && npm test     # tsc + node --test  → 12 tests
```

There are no network or model calls in the tests — they run against the committed dataset and
assert real behaviour (e.g. "greenest own-share ≥ cheapest", "soonest start ≤ cheapest",
"tasks stagger across the solar plateau", "savings are positive for an overnight EV charge").

## Deployment

**Backend — fully automated.** Push to `main` triggers `.github/workflows/deploy.yml`, which
SSHes to `getfletcher.ai`, fast-forwards the repo, runs `npm ci && npm run build`, restarts the
`reonic-backend.service` systemd unit, and smoke-tests `https://getfletcher.ai/api/health`.
One-time server provisioning: **[deploy/SERVER_SETUP.md](deploy/SERVER_SETUP.md)**.

**iOS — TestFlight.** Archive + upload via Xcode or the scripted path:
**[deploy/TESTFLIGHT.md](deploy/TESTFLIGHT.md)**.
