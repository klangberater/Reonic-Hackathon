# Lumen — technical documentation

A deep-dive for evaluation: how the system is built, the algorithms behind each claim, the
tools and frameworks used, and — explicitly — **what is real computation vs. what is simulated
data**. Companion to the [README](../README.md) and [API reference](API.md).

---

## 1. Design principles

1. **Tools own the numbers, the LLM owns the words.** Every figure shown or spoken — savings,
   solar share, schedules, the anomaly verdict — is the deterministic return value of a
   function over real data. The language model only narrates; it never invents a number.
2. **One coherent dataset, no database.** The source of truth is a physically-balanced year of
   15-minute telemetry (JSON). The only mutable runtime state is an in-memory ledger of
   scheduled loads. This keeps the system inspectable and the demo reproducible.
3. **Source-first, not price-first.** The planner classifies energy by where it comes from —
   **free solar → home battery → paid grid** — because the product story is self-consumption,
   not arbitrage. Cost is the tie-breaker, not the goal.
4. **Honest under inspection.** It runs on a real wall clock against real telemetry; numbers
   change with the time of day and are defensible. Nothing is hard-coded for a screenshot.

## 3. The dataset and its transform pipeline

**Raw input:** `enpal-track-dataset/` — the organizer's illustrative dataset (never modified).
**Output:** `data/` — a physically-coherent, demo-ready dataset, regenerated by
`scripts/transform_dataset.py` (Python 3). Full description: [data/README.md](../data/README.md).

### Why a transform is needed
The raw data's `outdoor_temp_c` and `heatpump_kw` channels are **seasonally inverted** relative
to PV/price/calendar — its "June" is a freezing, heat-pump-heavy winter that happens to be
sunny. Left as-is, the home screen would show nonsense (heatwave + blasting heat pump). The
transform rebuilds the physically-driven channels from scratch so the entire year is coherent.

### What it does (per household)
1. **Year-shift 2025 → 2026**, landing the data's June 20/21 on the demo weekend.
2. **Rebuild `outdoor_temp_c`** as a seasonal + diurnal curve (cold January, warm July) with
   per-city offsets.
3. **Overlay the real Munich forecast** (`munich_forecast_2026-06-20.json`, pulled from
   **Open-Meteo**) onto the hero home's now-window, so the on-screen temperature matches what
   is literally outside during judging.
4. **Recompute `heatpump_kw`** from temperature (heavy when cold, ~0 in summer; per-home cap;
   0 for the no-heat-pump home).
5. **Inject a winter heat-pump anomaly** (~+60%) for HH-1001 in a genuinely cold week
   (2026-01-12…19) — a real, detectable fault to drive the proactive-nudge demo.
6. **Recompute `total_consumption_kw`** and **re-run a greedy self-consumption battery+grid
   dispatch**, so the energy balance `pv + import + discharge = consumption + export + charge`
   holds to floating-point precision at every one of the 35,040 steps.
7. **Recompute `monthly_bills.json`**, realign `insight_events.json`, and year-shift
   `dynamic_prices.json` + `contracts.json`.

PV, house load, EV charging, and price are kept from the raw data (those already track the
calendar correctly). Result: max energy-balance error across 4 homes × 35,040 steps ≈ 1e-15 kW.

### Dataset shape
- 4 households (`HH-1001`…`HH-1004`); HH-1001 = Familie Becker, Munich (9.8 kWp PV, 10 kWh
  battery, heat pump, 60 kWh EV) is the hero.
- Per-record fields: `timestamp, outdoor_temp_c, pv_production_kw, house_load_kw, heatpump_kw,
  ev_charging_kw, total_consumption_kw, battery_charge_kw, battery_discharge_kw, battery_soc_kwh,
  battery_soc_pct, grid_import_kw, grid_export_kw, price_eur_per_kwh`.

`data.ts` lazy-loads each ~13 MB timeseries once and builds a `timestamp → index` map for O(1)
lookup.

## 4. The planner — `optimizeLoad.ts`

Finds the best window to run **one** flexible load over a search horizon (default 24 h, or up
to a deadline). The model is deliberately simple and defensible.

### Energy accounting (per 15-min slot in a candidate window)
- **Free solar** = the solar currently exported to grid (`grid_export_kw`) minus what
  already-committed loads claim. The load takes solar first.
- **Battery** = the shortfall is drawn from stored solar (battery SoC at window start), capped
  by battery power.
- **Grid** = whatever remains, priced at that slot's retail price.

So each window yields `{ free, battery, grid (kWh), cost (€) }`. **Source** is derived:
`free` if grid < 2% of the load, `partial` if some own energy, else `paid`. **Own-share %** =
`(free + battery) / total`.

### Sequential routing (multi-load coordination)
Committed loads (in the ledger) claim their solar/battery first, so the *next* load sees what's
left and routes around them. This is how several tasks coordinate without a global solver.

### Three objectives (the Cheapest / Greenest / Soonest toggle)
The window search is one loop; only the selection rule differs:
- **cheapest** — minimise grid cost; tie-break: less overlap with other loads (see staggering),
  then earliest.
- **greenest** — maximise own energy (`free + battery`); tie-break: less overlap, then cheaper.
- **soonest** — earliest feasible start; deliberately no staggering ("just run it now").

### Staggering (why the plan looks like a real schedule)
At solar noon every window is "free", so a naive planner piles every task onto the earliest
slot ("everything at 13:00", deadline/mode irrelevant). The cheapest/greenest tie-break
therefore prefers the window overlapping the **fewest** already-committed loads. Tasks spread
across the solar plateau into a readable schedule **without ever trading cost for spread** —
staggering only breaks exact ties. `soonest` keeps cramming early, so the modes visibly differ.

### Three surfaces from one call
`optimizeLoad` also returns a 24-h **ribbon** (each hour coloured by source) and per-hour
**slots** (feasibility, source, own-share, grid cost) for the interactive picker — so the
device sheet, the recommendation, and the drag-to-choose timeline all come from one request.

## 5. The day planner — `planDay.ts`

The "Plan my day" engine. Schedules several tasks at once and returns the day's solar curve
plus aggregate savings. **Computes and commits** in one call (idempotent — re-plan/mode/nudge
re-call it; the ledger replaces per device).

1. Remove prior commitments for exactly the requested devices (clean re-plan).
2. Order tasks: pinned (nudged) first, then by deadline ascending.
3. For each task: build the device (EV is sized from its charge target —
   `energy = capacity × (target − 20)/100`), run `optimizeLoad` against the current ledger in
   the chosen mode, commit the winning window so the next task routes around it.
4. **Solar curve:** `pv_production_kw` for hours 06–23 of the now-day (18 points) for the hero
   timeline background.
5. **Savings (honest baseline):** each task's "naive" alternative is **running it last-minute,
   finishing exactly at its deadline** (car charging overnight to be ready by 7 am, appliance
   in the evening). `savedEur` = Σ(baseline cost − plan cost), floored at 0; `savedCo2Kg` =
   Σ(baseline grid kWh − plan grid kWh) × 0.40 (German grid factor). This keeps the headline
   non-zero and meaningful even at solar noon, where a "vs run it right now" baseline would be
   ~€0. `solarSharePct` = Σ own kWh / Σ total kWh.

## 6. Anomaly detection — `anomaly.ts`

The math behind the "Needs a look" card and the assistant's heat-pump answer. It produces
**evidence only** (no prose); the causal narrative is the model's job.

- Aggregate daily heat-pump kWh, house kWh, and mean outdoor temperature over a trailing
  21-day window from "now".
- Build the home's **own** temperature→consumption baseline (median heat-pump kWh per 2 °C
  bucket) — i.e. weather-normalise against the house's own history, not a generic model.
- A day is anomalous when observed draw exceeds the temperature-expected draw by ≥ 40%; flag a
  run of ≥ 3 sustained anomalous days that is still active (ends within 3 days of now).
- Output: observed vs expected kW, % over, day count, temperature range, and **whether other
  loads are normal** (is the excess isolated to the heat pump?), plus a deterministic one-line
  summary for the card.

This is the project's "genuine AI understanding" claim: **detection is real math; the cause is
LLM reasoning** over that evidence. Verified live — HH-1001 winter shows +64% (2.2 vs 1.4 kW);
summer and the no-heat-pump home show no false positive.

## 7. Grounded assistant — `openaiChat.ts`

A natural-language layer that is structurally prevented from hallucinating figures.

- **Loop:** OpenAI Chat Completions (`gpt-4o`, `tool_choice: auto`, ≤ 6 turns). The model
  calls tools; the backend executes them against the data layer; results are fed back; the
  model writes a 1–3 sentence plain answer.
- **Tools** (bound to the current home + "now"): `get_now`, `get_money`, `get_household`,
  `get_devices`, `get_insights`, `list_commitments`, `optimize_load(device, deadline?)`,
  `explain_anomaly`. These are the *same* read models the app uses.
- **System prompt rules** enforce grounding: always call a tool for any number/time/price/
  status; never invent figures; interpret a net-positive month as a credit ("you earn €X", not
  "costs €X"); for "why is my heat pump/bill high" call `explain_anomaly` and **never blame the
  weather when the weather-normalised evidence rules it out** — name the likely cause (defrost
  fault / low refrigerant / thermostat) and suggest a service check.
- **Response** includes `toolsUsed` so the UI (and reviewers) can see which computations backed
  the answer. Requires `OPENAI_API_KEY`; absent → `503` (the rest of the app is unaffected).

> The hackathon is OpenAI-sponsored, so the assistant uses OpenAI. Everything else (planner,
> anomaly, savings) is provider-independent deterministic code.

## 8. The real-time clock — `clock.ts`

`resolveNow(clock?, at?)` decides what "now" means:
- `at` set → use it verbatim (scripted demos / tests).
- `winter` → fixed `2026-01-15T08:00` (a cold morning that surfaces the heat-pump anomaly).
- otherwise (`summer`/default) → **live**: the real Europe/Berlin wall clock, floored to the
  15-minute grid and **pinned to the data year (2026)** so the timestamp always exists in the
  dataset (which spans all of 2026). The iOS `ClockStore`/`PlanDayViewModel` mirror this for
  deadline anchoring.

Effect: the home screen reflects the actual time of day (evening shows winding-down solar; the
plan defers loads to real overnight / next-day sun), while `winter` remains a stable, scripted
scenario for the anomaly demo.

## 9. iOS architecture

SwiftUI, iOS 17, **no third-party packages**. Project generated by **XcodeGen** from
`ios/project.yml` (the `.xcodeproj` is gitignored — run `xcodegen generate` after pulling).

- **`RootPager`** — two start screens swiped horizontally via an iOS-17 paging `ScrollView`
  (not a `.page` `TabView`, whose `UIPageViewController` swallows the top safe-area inset and
  slides content under the Dynamic Island). The pager re-insets pages by the true safe-area
  insets so headers clear the camera and the ask bar clears the home indicator.
- **`ClockStore`** — a single `ObservableObject` holding the demo clock, injected into both
  screens so changing it (Settings) updates Home and Plan together and resets the ledger.
- **MVVM** — `HomeViewModel` / `PlanDayViewModel` (`@MainActor`) own state and call `APIClient`
  (async `URLSession`, `convertFromSnakeCase` decoding).
- **`DayTimeline`** — the hero timeline is a hand-drawn SwiftUI `Canvas`/`Path`: solar-curve
  area + lane-packed task blocks, each split-shaded green (own share) vs grey (grid).
- **`Theme`** — semantic, light/dark-adaptive design tokens; `source(_:)` maps free/partial/
  paid to colour everywhere.
- **`NotificationManager`** — local reminders for non-controllable appliances.

Key files: `LumenApp`, `RootPager`, `HomeView(+ViewModel)`, `PlanDayView(+ViewModel)`,
`DayTimeline`, `DeviceSheetView`, `FlowDetailView`, `ChatView`, `SettingsView`, `Models`,
`APIClient`, `Config`, `Theme`, `ClockStore`, `NotificationManager`. See
[ios/README.md](../ios/README.md).

## 11. Testing

`backend/*.test.ts` run via `npm test` (`tsc` + `node --test`, 12 tests), against the committed
dataset with no network/model calls. They assert real planner behaviour, e.g.:
- objective invariants — greenest own-share ≥ cheapest; soonest start ≤ cheapest;
- devices — all six task types present for a heat-pump + EV home; EV energy scales from target;
- day plan — 18-point curve, non-negative savings, tasks route around each other;
- **staggering** — cheapest spreads equally-free tasks to distinct starts; soonest crams;
- **savings** — charging the car by 7 am beats the overnight last-minute baseline (> 0).

iOS logic that can't be unit-tested headlessly (paging, timeline, deadline rollover) was
verified by building and screenshotting the simulator across normal/large Dynamic Type.

## 13. What's real vs. simulated (for evaluation)

| Aspect | Status |
|--------|--------|
| Energy telemetry | **Simulated but physically coherent** — derived from the organizer dataset; energy balance holds to ~1e-15 at every step. Outdoor temperature for the hero now-window is a **real** Open-Meteo Munich forecast. |
| "Now" | **Real wall clock** (default), pinned to the data year so it indexes real telemetry. |
| Planner, savings, solar-share, schedule | **Real deterministic computation** over the telemetry. |
| Anomaly detection | **Real** weather-normalised statistics over the home's own history. |
| Money forecast | **Real** month-to-date + projection from the dispatch + tariff. |
| Assistant prose | **LLM-generated**, but every number in it is a tool's computed return value. |
| Device automation | **Simulated** — controllable loads (EV/heat boosts) would drive a wallbox/heat-pump API in production; appliances become reminders. The ledger models the committed schedule. |

The honest summary: the *intelligence* (planning, anomaly reasoning, savings) is genuine code
on a coherent dataset and a live clock; the *telemetry and actuation* are simulated, as they
must be for a hackathon without a connected home.
